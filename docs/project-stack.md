# Текущий стек и подходы проекта

## Кратко

Проект представляет собой калькулятор коммунальных платежей с хранением помесячных показаний счетчиков. В репозитории одновременно присутствуют:

- текущая web-версия для схемы `GitHub -> Netlify -> Supabase`
- более ранняя локальная Python-реализация с файловым хранением

Основным актуальным стеком выглядит именно web/serverless-вариант, потому что он описан в `README.md`, настроен через `netlify.toml` и использует `Supabase Postgres` как постоянное хранилище.

## Актуальный production-стек

### Frontend

- Статический фронтенд без фреймворка
- `HTML + CSS + vanilla JavaScript`
- Подключение клиентского кода как `ES modules`
- Адаптивная верстка через CSS Grid и media queries
- UI-стилизация в духе `Material Design 3`

Ключевые файлы:

- `web/index.html`
- `web/styles.css`
- `web/app.js`

### Backend

- `Netlify Functions` на `Node.js 20`
- Формат функций: `.mjs`
- Серверная логика разделена на entrypoint-функции и общие модули в `netlify/functions/_lib`

Ключевые файлы:

- `netlify/functions/month.mjs`
- `netlify/functions/calculate.mjs`
- `netlify/functions/_lib/calculations.mjs`
- `netlify/functions/_lib/storage.mjs`

### База данных и инфраструктура

- `Supabase` как BaaS
- `Postgres` в Supabase для хранения показаний
- Доступ к БД с сервера через `@supabase/supabase-js`
- Развертывание через `Netlify`
- Исходники хранятся в `GitHub`

Ключевые файлы:

- `supabase/schema.sql`
- `netlify.toml`
- `package.json`

## Зависимости и окружение

### Node.js / npm

В `package.json` зафиксированы:

- runtime-зависимость: `@supabase/supabase-js`
- dev-зависимость: `netlify-cli`

Сборочного шага по сути нет:

- `build`: `echo "No build step required"`

Это означает, что проект сейчас опирается на максимально простой pipeline без bundler'а и без frontend-framework tooling.

### Переменные окружения

Серверный слой использует:

- `SUPABASE_URL`
- `SUPABASE_SERVICE_ROLE_KEY`

Секретный ключ применяется только внутри Netlify Functions и не передается на клиент.

## Архитектурные подходы

### 1. Разделение на тонкий клиент и serverless-API

Фронтенд отвечает за:

- рендер формы
- выбор периода
- загрузку данных месяца
- отправку данных на расчет
- отображение результата и сообщений

Сервер отвечает за:

- валидацию входных значений
- сохранение показаний
- получение предыдущего месяца
- расчет итогов
- работу с Supabase

Подход хороший тем, что клиент остается простым, а критичная логика и доступ к БД вынесены на сервер.

### 2. Выделение доменной логики в отдельный модуль

Расчетная логика вынесена в `netlify/functions/_lib/calculations.mjs`:

- нормализация ключа месяца
- вычисление предыдущего месяца
- парсинг тарифов и показаний
- вычисление дельт
- проверка на отрицательный расход
- расчет суммы по воде и электричеству

Это признак модульного подхода: HTTP-обработчики не перегружены бизнес-логикой.

### 3. Хранение по месячному ключу

Основной идентификатор записи: `month_key` в формате `YYYY-MM`.

Это упрощает:

- upsert-операции
- чтение по месяцу
- поиск предыдущего периода
- понятную связь UI с БД

### 4. Идемпотентное сохранение

Сохранение реализовано через `upsert` по `month_key`.

Практический эффект:

- повторное сохранение месяца обновляет запись, а не создает дубликат
- форма может использоваться для корректировки данных за месяц

### 5. Seed-подход для стартовых месяцев

В `netlify/functions/_lib/seed-data.mjs` зашиты стартовые данные за январь и февраль 2026, а `storage.mjs` умеет автоматически записывать их в БД при первом обращении.

Это упрощает первый запуск и уменьшает количество ручных подготовительных шагов.

## Подходы к данным

### Модель хранения

Таблица `public.meter_readings` хранит:

- год и месяц расчета
- показания холодной воды
- показания горячей воды
- показания электроэнергии по зонам `T1/T2/T3`
- `created_at`
- `updated_at`

В БД есть:

- `primary key` по `month_key`
- `check constraints` на год и месяц
- триггер на автоматическое обновление `updated_at`

### Формула расчета

Применяется модель:

- расход = текущие показания - показания предыдущего месяца
- вода = холодная + горячая + водоотведение
- электричество = сумма по тарифам `T1/T2/T3`

Система запрещает отрицательные дельты, то есть защищает расчет от ситуации, когда текущие показания меньше предыдущих.

## Frontend-подходы

Во фронтенде используются следующие практики:

- работа через единый объект `state`
- явное кеширование DOM-элементов в `elements`
- разделение функций на `render`, `load`, `submit`, `init`
- отсутствие сторонних UI-библиотек
- минималистичная интерактивность без сложного состояния

Сетевое взаимодействие построено через `fetch` к двум endpoint'ам:

- `GET /api/month`
- `POST /api/calculate`

## Backend-подходы

Serverless-слой использует:

- небольшие специализированные endpoints
- единый JSON-ответ с `statusCode`, `headers`, `body`
- локальную функцию-обертку `json(...)`
- обработку ошибок через `try/catch`
- отсутствие ORM и прямую работу через Supabase client

Это простой и прозрачный подход без лишней абстракции.

## Наследие и эволюция проекта

В репозитории сохранена Python-версия:

- `app.py`
- `calculator.py`
- `storage.py`
- `meter_history.json`

Ее особенности:

- локальный HTTP-сервер на `wsgiref`
- серверный HTML-рендеринг
- файловое хранение в JSON
- аналогичная доменная модель расчета через `dataclass`

По структуре видно, что текущая JS/serverless-реализация унаследовала ключевую бизнес-логику из Python-версии, но сменила:

- способ доставки UI
- способ хранения данных
- серверную платформу

## Что можно считать текущими инженерными практиками проекта

- Простая архитектура без тяжелого фреймворка
- Явное разделение UI, бизнес-логики и доступа к данным
- Serverless-first подход для backend-части
- BaaS-first подход для персистентности
- Конфигурация через environment variables
- Идемпотентная запись данных
- Валидация на клиенте и на сервере
- Поддержка ручного редактирования тарифов в UI
- Наличие базовой инициализации данных через seed

## Текущие ограничения, которые тоже важно зафиксировать

- Автотестов в репозитории сейчас не видно
- Нет линтера, форматтера и явного quality gate
- Нет TypeScript, поэтому типизация в JS-части отсутствует
- Нет frontend-framework'а и переиспользуемых UI-компонентов
- Тарифы не вынесены в отдельную таблицу или конфигурационное хранилище
- Доменная логика продублирована в Python и JavaScript

## Итог

Сейчас проект — это небольшой full-stack web-сервис на связке `vanilla JS + Netlify Functions + Supabase`, построенный вокруг простой доменной модели и serverless-подхода. Архитектура прагматичная: без сложной инфраструктуры, с понятным разделением ответственности и с фокусом на быстрый деплой и простую поддержку.
